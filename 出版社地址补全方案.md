# 出版社默认地址自动补全方案

> 生成时间：2026-06-20
> 背景：豆瓣图书原文无"出版地"字段，parser 解析后 publishPlace 永远为空，用户需手动查"这个社在哪"再填。
> 数据依据：用户导出的 265 条历史记录中 59 条图书，提取出的常用出版社频次表。

---

## 一、问题证据

### 1.1 parser 代码证据
`src/lib/parser.ts:102-103`（豆瓣解析函数 parseDoubanBook）：
```ts
// 出版社
result.publisher = fields['出版社'] || fields['出 版 社'] || '';
// ↑ 只填 publisher，publishPlace 完全没赋值
```

### 1.2 数据证据
导出的 59 条图书里：
- **57 条有地址** → 用户手动补的（豆瓣原文没有"出版地"字段）
- **2 条无地址** → 用户漏补了，分别是 University of North Carolina Press（应为 Chapel Hill）

### 1.3 用户痛点
每粘贴一本豆瓣书，都要：① 查出版社在哪 → ② 切到地址字段 → ③ 手动填。
按 59 条数据里 25 个不同出版社算，高频就那 10 来个，完全可建表自动化。

---

## 二、方案设计

### 2.1 核心思路
建一份「出版社 → 出版地」映射表，在两个时机自动补全：
1. **parser 解析后**：解析豆瓣/GB-T 格式时，若 publishPlace 为空，按 publisher 查表补
2. **表单 onChange**：用户在 ManualForm 改出版社字段时，若地址为空，按新出版社查表补

**铁律**：只在 `publishPlace` 为空时补，**绝不覆盖用户已填的地址**。

```ts
/**
 * 按出版社名查默认出版地
 * 匹配策略：先精确，再包含（如"生活·读书·新知三联书店"匹配"三联书店"）
 * 返回 undefined 表示无匹配
 */
export function lookupPlace(publisher: string): string | undefined {
  if (!publisher || !publisher.trim()) return undefined
  const p = publisher.trim()

  // 1. 精确匹配
  if (PUBLISHER_PLACES[p]) return PUBLISHER_PLACES[p]

  // 2. 包含匹配（去掉空格和标点后再比，避免"出 版 社"这种干扰）
  const normalize = (s: string) => s.replace(/[\s·、，,]/g, '')
  const np = normalize(p)
  for (const [key, val] of Object.entries(PUBLISHER_PLACES)) {
    if (normalize(key) === np) return val
  }

  // 3. 子串包含（用户输入"生活·读书·新知三联书店"含"三联书店"）
  //    只在 key 较短（≤6字）时启用，避免"出版社"误匹配一堆
  for (const [key, val] of Object.entries(PUBLISHER_PLACES)) {
    if (key.length <= 6 && p.includes(key)) return val
    if (key.length <= 6 && key.includes(p)) return val
  }

  return undefined
}
```

### 2.3 触发点 1：parser 解析后补全

修改 `src/lib/parser.ts`，在 `parseCitationText` 顶层返回前加一段：

```ts
import { lookupPlace } from './publisher-places'

export function parseCitationText(text: string): Partial<Citation> & { id: string } {
  // ...原有逻辑...

  // 解析完成后，若图书类型且地址为空，尝试补全
  if (result.type === 'book' && !result.publishPlace && result.publisher) {
    const place = lookupPlace(result.publisher)
    if (place) result.publishPlace = place
  }

  return result
}
```

**注意**：只在 parseCitationText 顶层补一处，不要在每个子解析器里都加。

### 2.4 触发点 2：表单 onChange 联动

修改 `src/components/ManualForm.tsx`，给出版社字段的 onChange 加联动：

```ts
const handlePublisherChange = (value: string) => {
  const newCitation = { ...citation, publisher: value }
  // 若地址为空或地址原本就是自动补的，按新出版社查表
  if (!newCitation.publishPlace || isAutoFilled) {
    const place = lookupPlace(value)
    if (place) {
      newCitation.publishPlace = place
      setIsAutoFilled(true)
    } else if (isAutoFilled) {
      // 之前是自动填的，现在出版社不认识了，清空
      newCitation.publishPlace = ''
      setIsAutoFilled(false)
    }
  }
  setCitation(newCitation)
}
```

**视觉提示（可选，P2 优先级）**：自动填的地址字段加浅黄色背景或小角标，用户能看出"这是系统猜的"。不做也行，因为用户改了就保留用户值。

### 2.5 不做的事

- ❌ 不做"出版社 IP 库"那种完整数据库——只覆盖老板常用 + 学术圈常见
- ❌ 不联网查出版社地址——离线映射够用，联网增加延迟和失败点
- ❌ 不覆盖用户已填地址——铁律
- ❌ 不强制填——查不到就留空，用户可手动补

## 三、验证用例

| 输入 publisher | 期望输出 publishPlace | 匹配方式 |
|----------------|----------------------|---------|
| `社会科学文献出版社` | `北京` | 精确 |
| `江苏人民出版社` | `南京` | 精确 |
| `生活·读书·新知三联书店` | `北京` | 包含（normalize 后精确） |
| `生活读书新知三联书店` | `北京` | normalize 后精确 |
| `三联书店` | `北京` | 精确 |
| `Cambridge University Press` | `Cambridge` | 精确 |
| `The University of North Carolina Press` | `Chapel Hill` | 精确（已加 The 前缀条目） |
| `中华书局` | `北京` | 精确 |
| `未知出版社` | `undefined` | 无匹配，留空 |
| `人民出版社` | `北京` | 精确（注意不能被"中国人民大学出版社"抢先匹配） |
| `中国人民大学出版社` | `北京` | 精确 |

**边界 case**：`人民出版社` vs `中国人民大学出版社` vs `北京大学出版社` vs `北京出版社`——
都用精确匹配优先，不会乱。子串匹配只对 ≤6 字的短 key 开启，且 `人民出版社`(5字) 是子串但精确匹配会先命中它自己。
